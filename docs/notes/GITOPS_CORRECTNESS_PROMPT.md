## SECTION D – Critical correctness / security bugs

### D1. AppProject namespaceResourceWhitelist has the wrong API group for AzureKeyVaultSecret

File: `deployments/appset/templates/project.yaml`

The chart whitelists:
```yaml
- group: ""
  kind: AzureKeyVaultSecret
```

But the CRD's actual API group is `spv.no` (see `deployments/charts/backend/templates/akvsecrets.yaml` and `deployments/charts/backend/templates/akvdocker.yaml`, both `apiVersion: spv.no/v2beta1`).

With `group: ""`, ArgoCD will reject every AzureKeyVaultSecret at sync time because the resource isn't in the whitelist. Fix:
```yaml
- group: spv.no
  kind: AzureKeyVaultSecret
```

### D2. AppSet destination namespace does not match AppProject destination

Files:
- `deployments/appset/templates/project.yaml`
- `deployments/appset/templates/appset.backend.yaml`
- `deployments/appset/templates/appset.frontend.yaml`

`project.yaml` allows only:
```yaml
destinations:
  - namespace: "{{ .Values.env }}-{{ .Values.appName }}"
```

But both `appset.backend.yaml` and `appset.frontend.yaml` set:
```yaml
destination:
  namespace: {{ .Values.env }}
```

Every Application generated by the AppSet will be rejected by the AppProject ("application destination is not permitted"). The runbook and the rest of the docs already assume `<env>-<appName>` namespaces (`docs/runbook.md` line ~38: `-n <env>-<app-name>`).

Fix both AppSet templates so the destination namespace is `{{ .Values.env }}-{{ .Values.appName }}` and update `tests/test_helm.sh` to assert that:
1. The AppProject `destinations.namespace` value
2. Both ApplicationSets' `template.spec.destination.namespace` value

are byte-equal to `<env>-<appName>` (use the test's `$ENV`/`$APP_NAME`).

### D3. Per-env tpls override secure-default volumes/volumeMounts to empty arrays

Chart defaults (`deployments/charts/backend/values.yaml`, `deployments/charts/frontend/values.yaml`) provide `emptyDir`-backed volumes at `/tmp` (and `/var/cache/nginx`, `/var/run` for frontend) so containers can run with `readOnlyRootFilesystem: true`. Every per-env tpl currently contains:

```yaml
volumes: []
volumeMounts: []
```

Helm does NOT deep-merge lists; an empty list `[]` **overrides** the chart default, so the rendered Deployment has no writable `/tmp`. Containers will CrashLoop on the first write under `readOnlyRootFilesystem: true`.

Fix: in all 12 per-env value tpls, **delete** the lines:
```
volumes: []
volumeMounts: []
```
(also `volumes: []` in `frontend/{dev,qa,stage,prod}/ui/values.yaml.tpl`).

Let the chart defaults flow through. Per-env files can still append by using a populated list.

Files to edit:
- `deployments/values/backend/{dev,qa,stage,prod}/api/values.yaml.tpl`
- `deployments/values/backend/{dev,qa,stage,prod}/auth/values.yaml.tpl`
- `deployments/values/frontend/{dev,qa,stage,prod}/ui/values.yaml.tpl`

### D4. test_charts.sh hard-fails when PyYAML is not installed

## SECTION F – Production-quality polish

### F1. PodDisruptionBudget default crashes node drain when replicaCount=1

`charts/backend/values.yaml` and `charts/frontend/values.yaml` default to:
```yaml
podDisruptionBudget:
  enabled: true
  minAvailable: 1
```

`replicaCount: 1` (dev, qa, single-replica auth) + `minAvailable: 1` means voluntary disruption (node drain, autoscaler scale-down) is forbidden — a known deadlock. Switch the chart default to `maxUnavailable: 1`, and update `templates/poddisruptionbudget.yaml` in BOTH charts to render `minAvailable` when set, otherwise `maxUnavailable`. Pattern:

```yaml
spec:
  {{- if .Values.podDisruptionBudget.minAvailable }}
  minAvailable: {{ .Values.podDisruptionBudget.minAvailable }}
  {{- else }}
  maxUnavailable: {{ .Values.podDisruptionBudget.maxUnavailable | default 1 }}
  {{- end }}
```

Per-env prod tpls (`replicaCount: 3`) can opt into `minAvailable: 2` if they want stronger guarantees — leave that as a per-env override, not a default.

### F2. apply-config.sh should refuse to run with CHANGE_ME placeholders

### F3. Workflow-level timeout-minutes on every ci-cd-*.tpl

Add `timeout-minutes:` at the job level in all four ci-cd-*.tpl files:
- dev/qa: `timeout-minutes: 15`
- stage: `timeout-minutes: 20`

This caps stuck `helm upgrade --wait` jobs at the workflow boundary even if the helm timeout flag is bypassed.

### F4. Wrap the Helm `--set` invocations against argument injection

In every `ci-cd-*.tpl`, the `Deploy AppSet` step builds `--set foo="${VAR}"` strings from environment variables sourced from `vars.*`. If a malicious environment variable contained shell metacharacters or commas, helm would mis-parse it. Switch multi-arg invocations to use `--set-string` for the user-controllable fields:

```bash
helm upgrade --install ... \
  --set-string appName="${APP_NAME}" \
  --set-string env="${ENV}" \
  --set-string org="${GITHUB_ORG}" \
  --set-string repo="${GITHUB_REPO}" \
  --set-string server="${CLUSTER_SERVER}" \
  --set-string IngressFrontendHost="${INGRESS_HOST}" \
  --set-string IngressBackendHost="${INGRESS_HOST}" \
  --wait --timeout 5m
```

`--set-string` always treats the RHS as a literal string and avoids helm's type coercion of values that look like numbers/booleans/lists.
